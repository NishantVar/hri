TL;DR

  • What this is: a human-driven end-to-end QA pass over RIView's website, daemon, and responder loop. Confirmed by source.
  • Scope: 27 scenarios across 5 areas — website-only state, storage invariants, daemon ↔ responder loop, validation, cross-cutting.
  • Before starting: run python3 -m pytest tests/. All 75 should pass.
  • What to track per scenario: step completion, pass/fail/skip status, free-form notes. Everything persists in this browser's localStorage.
  • Auto-coverage badges: green = existing test in tests/test_daemon.py; amber = TODO test gap flagged in the plan.
  • Dynamic loop scenarios (C1–C7): can be driven by a second terminal simulating the responder, or by the real riview-respond agent.
  • Cleanup: kill $DAEMON_PID && rm -rf .riview/ at the end of the run.

setup One-time setup

Get the daemon running, a clean session registered, and a browser tab open before starting section A.

SET Boot daemon and register a fresh sample session pending

Steps

  1. cd $REPO (your RIView checkout).
  2. rm -rf .riview/ — start clean.
  3. git status — confirm .riview/ is gitignored (no entries listed).
  4. python3 -m pytest tests/ — baseline green.
  5. Boot the daemon in the background: 
    python3 scripts/riview.py daemon &
DAEMON_PID=$!
sleep 1
  6. Register the sample spec: python3 scripts/riview.py submit sample. Capture <sid> from output.
  7. Open http://127.0.0.1:7891/sessions/<sid> in a browser tab.

Mid-run resets

To wipe only the overlay (keep revisions): rm -rf .riview/sessions/<sid>/reviews/*/review.json.

To tear down: kill $DAEMON_PID && rm -rf .riview/.

A Website-only state

No agent in the loop. Pure submit / reload / overlay-merge semantics in the browser ↔ daemon path.

A1Submit-then-reload persistence (happy path)pending

Steps

  1. Open /sessions/<sid>.
  2. Pick a decision card. Change status ai-confident → confirmed. Type comment "QA A1".
  3. Click Submit on that card. Watch for the success indicator.
  4. Hard reload (Cmd+Shift+R).

Expected

  • Status select shows confirmed. Comment textarea shows "QA A1".
  • Card is untouched (border = status color, Submit button disabled).
  • cat .riview/sessions/<sid>/reviews/1/review.json shows the entry with new_status: confirmed and comment: "QA A1".
Auto-coverage: test_session_page_renders_overlay_status_and_comment

Steps

Steps

  2. Reload. Confirm pre-fill + untouched.

C Daemon ↔ Responder loop

Simulate the responder with a second terminal that re-submits the spec (advancing revision), or run the real riview-respond agent against the session.

C1Fresh revision shows empty overlaypending

Steps

  1. From the A1 state (revision 1 has an overlay), responder writes a new revision: 
    # Edit sample/spec.md to change something canonical
python3 scripts/riview.py submit sample --session <sid>
  2. Reload the browser tab on /sessions/<sid>.

Expected

  • Page now shows revision 2 (check <script id="submit-config"> for base_revision: 2).
  • All cards canonical: no overlay borders, comments empty, statuses match the new spec.
  • No reviews/2/review.json exists.
Auto-coverage: partial — test_session_page_with_no_overlay_renders_empty_overlay_comments

C2Submit while responder watches /waitpending

Steps

  1. In a second terminal: python3 scripts/riview.py wait <sid>.
  2. In the browser, edit and submit a card on revision 2.
  3. Watch the wait call return.

Expected

C3Stale base, no node change (clean accept)pending

Steps

  1. On revision 2, open the page (base_revision=2). Edit a comment, but do not submit yet.
  2. In another terminal, responder rolls forward to revision 3 without changing the node you're editing: 
    # Edit a different node's body in sample/spec.md
python3 scripts/riview.py submit sample --session <sid>
  3. Submit your card (still on the stale base_revision=2 page).
  4. Reload.

Expected

  • Submit response: accepted: [{node_id: <yours>}], conflicts: [], current_revision: 3.
  • After reload, the page is now on revision 3 with the overlay merged in.
Auto-coverage: test_stale_submit_accepts_unchanged_node_conflicts_changed_node, test_stale_submit_with_base_equal_current_is_clean_accept

C7Responder pulls applied review, advances cleanlypending

Steps

  1. Submit a multi-card review on the current revision.
  2. Simulate the responder loop: 
    python3 scripts/riview.py pull <sid> > /tmp/qa-pull.json

D Validation & error paths

Hand-rolled curl probes against the daemon. Set TOKEN=$(cat .riview/token) first.

D1Malformed cleared_fields → 400pending

Steps

  1. Run the curl below, then repeat with cleared_fields: ["unknown_field"], cleared_fields: [123], cleared_fields: null: 
    curl -i -X POST http://127.0.0.1:7891/sessions/<sid>/review \
  -H "X-Riview-Token: $TOKEN" -H "Content-Type: application/json" \
  -d '{"spec_id":"<id>","spec_version":1,"base_revision":1,
       "reviews":[{"node_id":"some-id","cleared_fields":"comment"}]}'

Expected

  • All four return HTTP 400 with a clear error message. No mutation on disk.
Auto-coverage: test_cleared_fields_rejects_malformed_values

E3Refresh during pending submitpending

Steps

  2. Edit a card. Click Submit. Before the response arrives, hard-reload.
  3. Wait for everything to settle. Inspect review.json and the UI.
