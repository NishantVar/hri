# Storage root defaults to `<riview_repo>/.riview/`

The daemon's storage root — the directory holding the auth token, the cross-project session inbox, and per-session revision/review state — defaults to `<riview_repo>/.riview/` (resolved from `scripts/riview.py`'s own location). `RIVIEW_HOME=<path>` still takes precedence for users who want the prior `~/.riview/` layout or any other location; the repo-local default just removes the need to set the env var to keep all of one workstation's review state in one place.

Why move it off `~/`: with the responder/reviewer split, the daemon is intended to be **one website front door for many agents working on different specs** in different consuming-project checkouts. Putting the storage next to the daemon's own source means a single `git clone` of RIView is the entire deployment unit — the binary, its config, its sessions, and its on-disk audit trail all live and ship together. Switching machines is `git clone` plus `python3 scripts/riview.py daemon`; nothing in `~/` needs to be migrated. The directory is `.gitignored` because session state is per-machine and not part of the project source.

Alternatives considered: (1) **Keep `~/.riview/`** as the default. Rejected — it forces users to remember which `$HOME` a session lives under when working across machines, and the daemon's "one inbox, many projects" model doesn't actually need `$HOME` placement (it works equally well repo-relative). (2) **Per-consuming-project storage** (each spec's daemon state in `<consuming_project>/.riview/`). Rejected by the user explicitly: one website fronting many projects is the point; sharding storage by project undoes that. (3) **System-wide `/var/lib/riview/`** like a packaged service. Rejected — RIView is a single-user dev tool, not a system service, and per-machine pathing under `~/` or the repo is the right granularity.

Migration: existing deployments keep working by exporting `RIVIEW_HOME=~/.riview` (or whatever path they were using). No data is copied automatically; the env-var override is the supported migration path.
