# Per-node stale-submit acceptance with fingerprint diff

A POST to `/sessions/<id>/review` must carry a `base_revision` (the revision number the reviewer's draft was rendered against). The daemon recomputes a per-node `fingerprint` from its own stored revision snapshots — for each node, `sha256` over the node's reviewer-visible fields (`kind`, `title`, `status`, `source_anchor`, `depends_on`, kind-specific fields, and `body_md` from the matching `<anchor>` in `source.md`) — both at `base_revision` and at the session's `current_revision`. Entries whose two fingerprints match are accepted into the current revision's review; entries that differ (or whose node didn't exist at `base_revision`) are returned in `conflicts` and never merged. The response is `{accepted, conflicts, base_revision, current_revision, review_count, event_seq}` so the client can show "N of M accepted; the following nodes moved on, please resolve" without re-fetching the whole session. Persisted reviews are normalized to the current revision (`spec_version` rewritten, `base_revision` stripped) so the on-disk review file is always coherent with the revision it lives under.

Alternatives considered: (1) session-level locking — reject the entire POST whenever `base_revision != current_revision`. Rejected because a reviewer who'd already typed a comment on an untouched node would lose their work to a totally unrelated change on another node, and the per-card "Submit decision" UX would become useless the moment a responder rolled a new revision. (2) All-or-nothing accept on fingerprint diff — accept the POST only if every entry's fingerprint matches. Rejected for the same reason: any single stale entry would block all the fresh ones, which defeats the whole point of letting a human and a responder agent work on different nodes concurrently. (3) Client-computed fingerprints sent with each entry — the daemon trusts whatever the browser ships. Rejected: it makes the trust boundary porous (any client can label any entry "fresh"), and it bloats the wire payload with a hash per entry that the server can recompute deterministically from data it already owns. Per-node fingerprinting on the server side is the smallest unit of optimistic concurrency that keeps both flows working without trusting the client.
