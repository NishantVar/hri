# The responder skill lives in the repo, installed via symlink

The `riview-respond` Claude Code skill lives at `<repo>/skills/riview-respond/` (alongside `scripts/` and `docs/`), not in a per-user `~/.claude/skills/` directory. Helper scripts under `tools/` resolve their imports via `$RIVIEW_REPO` so the skill is location-independent. Install pattern: `ln -s "$RIVIEW_REPO/skills/riview-respond" ~/.claude/skills/riview-respond`.

Alternatives considered: (1) skill in `~/.claude/skills/` only — the Claude Code default. Rejected because the skill's contract is the daemon's wire protocol (`base_revision`, fingerprint diff response shape, `/wait` semantics, session meta format); the two evolve together and a per-user copy drifts the moment either side changes. Keeping the skill in the repo makes daemon changes and skill changes a single reviewable diff. (2) Skill as a published package (pypi / npm / a separate git repo) — overbuilt for a single-user local tool. It adds a release process, a versioning story, and a "what version of the skill am I running against this daemon?" support burden, all to ship a directory of text files that already sit next to the daemon they talk to. Note: this is about distribution overhead, not ADR-0001 — packaging the skill itself wouldn't introduce a Python runtime dependency. (3) Install by `cp -r` instead of symlink — works, but every daemon update silently strands the user's copy of the skill until they remember to re-copy.

The symlink install keeps the user's `~/.claude/skills/` directory in the loop (so Claude Code discovers the skill), while the canonical source stays under git history with the daemon. The skill is model-agnostic (Claude Code today, Codex / Gemini tomorrow); the install pattern is identical across hosts that look in `~/.claude/skills/`.
