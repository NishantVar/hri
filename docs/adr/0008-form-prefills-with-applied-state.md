# Form prefills each card from the node's current applied state

On render, every card's status `<select>` is prefilled with the node's current applied `status` (read from the freshly-rendered `SPEC`), and an ambiguity card's resolution radio + freeform textarea is prefilled with the node's current applied `resolution` (the `{choice_id, freeform}` pair from the sidecar). The status dropdown includes the initial-state options (`ai-confident` for decisions, `open` for ambiguities and risks) so the prefilled option actually exists in the menu — there is no "— No change —" placeholder. The `comment` textarea is deliberately blank each cycle: comments are per-review-pass notes, not persistent node state, and pre-filling them with the previous cycle's comment would invite the reviewer to ship a stale note. There is no body-edit UI in the rendered form today — `body_edit` exists in the schema and the apply path, but a body-editing widget would need its own design pass (see PLANNED). Touched-detection diffs the live form against `APPLIED_BY_ID` (the snapshot built from `SPEC.nodes` at init), not against the form's initial DOM, so re-selecting the status the spec already has does not mark the card touched. `resyncCardDom(nid)` is the single source of DOM truth — called at init and again after a successful per-card submit — so the post-submit reset matches the same logic as the first render.

Alternative considered: a blank placeholder default ("— No change —") with `null` meaning "leave this node alone" — the obvious web-form pattern. Rejected because a RIView spec already *has* a defined state for every node, and the reviewer's job is to push that state forward, not to fill in a blank ballot. With a blank default the reviewer has to remember what the spec currently says for each card before deciding whether to keep it or change it, which is exactly the information the renderer already has. Prefilling also makes the per-card "Submit decision" button do the obviously-right thing on a card the reviewer didn't touch (no-op, dropped as empty on the server) rather than overwriting state with a placeholder.

Note: this is distinct from draft prefill (ADR-0009). The current-applied-state prefill is what you see on a fresh page load with no localStorage draft for this `base_revision`; the draft layer hydrates over it when one exists.
