# Render-time merge of the submitted review overlay

When the daemon serves `GET /sessions/<sid>`, it reads the current revision's submitted review (`reviews/<current>/review.json` — the "overlay") and pre-merges those entries into the in-memory spec before handing it to `render.build_html`. Submitted `new_status` lands on the node's `status` field; submitted `resolution` lands on `node.resolution`; submitted `body_edit` replaces the body between the node's anchor comments. The full per-node overlay entries are also emitted into the page as a sibling JSON island (`<script id="overlay-entries">{ <node_id>: { new_status?, resolution?, body_edit?, comment? }, … }`). The page's `APPLIED_BY_ID` baseline takes `comment` from that island, and `state[n.id].comment` is initialized from it so the textarea pre-fills on every reload. Touched-vs-applied diffing (`isTouchedNode`) compares the textarea against this baseline, so a pristine reload of submitted work shows the card as un-touched and Submit is disabled until the reviewer actually changes something.

The reason the renderer needs the full overlay entries (not just comments): the daemon's same-revision merge (`_merge_reviews_by_node_id`, ADR-0005) replaces the prior entry for a node by `node_id`. A sparse "only this field changed" submit would silently drop every other already-submitted field — e.g. editing only the comment after a prior `confirmed` + `body_edit` submit would leave the stored overlay holding just the comment, and a reload would snap the status back to canonical. So `buildEntryForNode` composes a **full effective overlay entry** for the submitted node: it starts from `OVERLAY_BY_ID[nid]` (preserving `new_status`, `resolution`, `body_edit` the reviewer did not touch) and layers the user's diffs vs `APPLIED_BY_ID` on top. After the daemon accepts the submission, `advanceBaselineFromEntry` updates `OVERLAY_BY_ID[nid]` and `APPLIED_BY_ID[nid]` in place so any subsequent edit on the same page composes against the new server state, not against the page-load snapshot.

One asymmetry the full-replace model can't express: clearing the *only* field of a comment-only overlay entry. The composed entry would be `{node_id}`, which the daemon's `is_empty_entry` filter drops before merge — so the stale comment would survive on reload. The fix is an explicit `cleared_fields: [...]` marker on the submitted entry: a list of overlay fields to drop. The daemon's overlay merge path treats it as a field-level diff against the prior entry (`(existing − cleared) ∪ incoming_non_clear_fields`); if only `node_id` remains, the node is evicted from the overlay entirely. The marker is scoped to the daemon's overlay merge path — `apply.py` is unchanged, since blank-comment / no-op semantics there are already handled by `is_empty_entry`. The client emits `cleared_fields` only when the composed entry would otherwise be empty but an overlay entry exists for that node; the normal partial-field case (e.g. clearing the comment when other overlay fields remain) just works via the full-replace contract.

The overlay is a render-time projection only — it is **not** written back into `node.review.comment` on the stored revision sidecar. That keeps the reviewer-comment baseline orthogonal to the applier's per-node `review` block, which carries the canonical applied-comment trail the responder will eventually write into the project spec. Mixing the two would either (a) re-submit stale comments on every reload as if they were new edits, or (b) force the daemon to maintain two separate "is this comment new?" diff rules — one for status/resolution (compare against applied baseline) and one for comments (compare against empty). Keeping comments as their own baseline lets the existing `isTouchedNode` check unify both.

Missing `reviews/<current>/review.json` (the typical case for a fresh revision the responder just rolled out) renders exactly as before — the overlay is the empty dict, `APPLIED_BY_ID[*].comment` is `""`, and the page draws the un-prefilled form. No file is materialized on render; the overlay file appears the moment the reviewer submits.

Alternatives considered: (1) **Server field-merge incoming entries into the stored overlay** instead of replace-merge, so the client could keep emitting sparse diffs. Rejected — that changes ADR-0005's by-node-id replace contract for all callers (CLI + agent reviews too), introduces an explicit "clear this field" tombstone problem, and forces the daemon to grow two diff disciplines (status/resolution against canonical, comments against the previous overlay value). The client-side full-entry contract keeps the merge semantics intact. (2) **Persist comments into `node.review.comment`** on each submit so the renderer doesn't need a separate baseline. Rejected — that's the applier's slot and would conflate reviewer-in-flight state with applier-confirmed state, breaking the responder's read contract. (3) **Re-fetch overlay via a separate XHR on page load instead of a JSON island.** Rejected — adds a network round-trip with no payoff on a localhost service, and means the first paint is briefly empty before the comment textareas backfill. (4) **Server-side full HTML rewrite of comment fields**, embedding the comment directly into the `<textarea>`'s initial value. Rejected — the renderer already hydrates the textarea via `resyncCardDom` from `state[node.id].comment`; threading the value through `APPLIED_BY_ID` keeps a single source of truth and avoids two prefill paths.
