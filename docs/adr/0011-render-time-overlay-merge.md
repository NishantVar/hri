# Render-time merge of the submitted review overlay

When the daemon serves `GET /sessions/<sid>`, it reads the current revision's submitted review (`reviews/<current>/review.json` — the "overlay") and pre-merges those entries into the in-memory spec before handing it to `render.build_html`. Submitted `new_status` lands on the node's `status` field; submitted `resolution` lands on `node.resolution`; submitted `body_edit` replaces the body between the node's anchor comments; submitted `comment` is returned separately and emitted into the page as a sibling JSON island (`<script id="overlay-comments">{ <node_id>: <comment>, … }`). The page's `APPLIED_BY_ID` baseline then includes a `comment` field sourced from that island, and `state[n.id].comment` is initialized from it so the textarea pre-fills on every reload. Touched-vs-applied diffing (`isTouchedNode`, `buildEntryForNode`) compares the textarea against this baseline, so a pristine reload of submitted work shows the card as un-touched and Submit is disabled until the reviewer actually changes something.

The overlay is a render-time projection only — it is **not** written back into `node.review.comment` on the stored revision sidecar. That keeps the reviewer-comment baseline orthogonal to the applier's per-node `review` block, which carries the canonical applied-comment trail the responder will eventually write into the project spec. Mixing the two would either (a) re-submit stale comments on every reload as if they were new edits, or (b) force the daemon to maintain two separate "is this comment new?" diff rules — one for status/resolution (compare against applied baseline) and one for comments (compare against empty). Keeping comments as their own baseline lets the existing `isTouchedNode` check unify both.

Missing `reviews/<current>/review.json` (the typical case for a fresh revision the responder just rolled out) renders exactly as before — the overlay is the empty dict, `APPLIED_BY_ID[*].comment` is `""`, and the page draws the un-prefilled form. No file is materialized on render; the overlay file appears the moment the reviewer submits.

Alternatives considered: (1) **Persist comments into `node.review.comment`** on each submit so the renderer doesn't need a separate baseline. Rejected — that's the applier's slot and would conflate reviewer-in-flight state with applier-confirmed state, breaking the responder's read contract. (2) **Re-fetch overlay via a separate XHR on page load instead of a JSON island.** Rejected — adds a network round-trip with no payoff on a localhost service, and means the first paint is briefly empty before the comment textareas backfill. (3) **Server-side full HTML rewrite of comment fields**, embedding the comment directly into the `<textarea>`'s initial value. Rejected — the renderer already hydrates the textarea via `resyncCardDom` from `state[node.id].comment`; threading the value through `APPLIED_BY_ID` keeps a single source of truth and avoids two prefill paths.
