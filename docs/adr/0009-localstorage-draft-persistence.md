# Per-revision sparse-diff draft persistence in localStorage

The form persists the reviewer's unsent edits to browser localStorage under `riview:draft:<sid>:<base_revision>` (daemon-served session pages) or `riview:draft:standalone:<spec_id>:<version>` (standalone-render pages). The stored value is a sparse diff against `APPLIED_BY_ID`: only entries the reviewer has *touched* (status/comment/resolution/body_edit diverges from the current applied state) are written. On load, the draft layer hydrates over the applied-state prefill from ADR-0008, so the form lands in "what I had typed last time" without ever having stored the bits the reviewer never edited. After a successful per-card or bulk submit, the corresponding entries are cleared from the draft. `DRAFT_RETAIN_REVS = 5` caps per-`sid` storage: when a new `base_revision` shows up, drafts for revision numbers older than `current - DRAFT_RETAIN_REVS + 1` are evicted — the storage retains keys for the latest five revision numbers per session.

Alternatives considered: (1) no persistence — close the tab, lose the work. The prior behavior, fine for the original "render once, submit once" mental model, but a regression as soon as the responder loop started rolling new revisions during a review session. (2) Per-session single key (no `base_revision` suffix) — simplest, but a new revision either silently wipes the reviewer's in-progress draft or replays stale entries against a fresh spec; both are hostile. (3) Full-form snapshot instead of sparse diff — would persist the spec's prefilled state alongside the reviewer's edits, conflating "I haven't touched this" with "I confirmed this matches"; also makes the draft useless to inspect (you can't tell at a glance what's actually changed). (4) `sessionStorage` instead of `localStorage` — survives tab reload but not tab close. Rejected because the realistic "I'll come back to this tomorrow" review flow is exactly what the persistence is for. (5) IndexedDB or a server-side draft — overbuilt for a single-user local tool whose actual persistence is git on submit.

The `base_revision`-keyed scheme means a responder agent rolling forward (current_revision N → N+1) does not invalidate the reviewer's in-progress draft for revision N; the reviewer can finish typing, submit against N (per-node fingerprint diff in ADR-0007 sorts out which entries still apply), and the next page load on N+1 starts a fresh draft slot. Eviction at five revisions keeps storage bounded without forcing the reviewer to play a fast game against the responder.
